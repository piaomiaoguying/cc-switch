# 4.6 画像整流器

## 概要

画像整流器は、API リクエストが上流プロバイダーに転送される前に、base64 画像ブロックを自動的に検出して置換するプロキシレイヤー機能です。

**解決する問題：** 多くのサードパーティ API プロバイダーはマルチモーダル（視覚）機能をサポートしていません。messages 配列に base64 データを含む `type: "image"` ブロックがあると、エラーが発生したり予期しない動作をします。

**解決策：** 整流器は転送前にリクエストを遮断し、すべての `messages[*].content` 配列をスキャンして `type: "image"` ブロックを見つけ、base64 データを削除し、指定されたスキルを呼び出して画像を処理するよう指示するテキストプロンプトに置き換えます。

## 有効化

1. CC Switch を開き、**設定**に移動します。
2. **プロキシ**タブに切り替えます。
3. **高度な整流器**までスクロールします。
4. **画像整流器**のスイッチをオンにします。
5. 必要に応じて、**スキル名**を変更します（デフォルト: `image-analysis`）。

## 設定項目

| 設定 | 説明 | デフォルト |
|---------|-------------|---------|
| 画像整流を有効化 | 機能のマスタースイッチ | オフ |
| スキル名 | モデルに呼び出すよう促すスキル識別子 | `image-analysis` |

## 動作の仕組み

1. プロキシはリクエスト転送前に画像整流が有効かどうかを確認します。
2. リクエストボディ内の各メッセージの `content` 配列を反復処理します。
3. `type` が `"image"` の各ブロックについて：
   - 削除された base64 データの文字数をカウント
   - 次のブロックが画像キャッシュ参照（`[Image: source: /path/to/file]`）かどうかを確認
   - 画像ブロックをテキストブロックに置換：`"[Image: この画像を読み取って分析するには '{skill_name}' スキルを使用してください。画像パス: /path/to/file.png]"`
4. 変更されたリクエストが上流プロバイダーに転送されます。

`type: "image"` ブロックがない場合、リクエストは変更されずに通過します。

## 画像分析スキルとの連携

整流器は[画像分析スキル](./4.7-image-analysis-skill.md)と連携するよう設計されています。両方を有効にすると：

1. 整流器が画像データを削除し、`image-analysis` を呼び出すプロンプトを挿入
2. モデルがテキストプロンプトを受け取り、`image-analysis` CLI コマンドを実行
3. スキルが事前設定された視覚 AI プロバイダーを使って実際の画像ファイル（またはクリップボードデータ）を処理
4. モデルが視覚分析結果を受け取り、応答を継続

## 注意事項

- 画像整流はプロキシ経由のリクエストにのみ影響します。直接の API 呼び出しには影響しません。
- デフォルトではオフになっており、マルチモーダル対応プロバイダーへの干渉を防ぎます。
- base64 画像データは数 MB になることがあり、削除するとリクエストボディサイズが大幅に削減されます。
